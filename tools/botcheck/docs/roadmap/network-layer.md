# Roadmap — network-layer fingerprinting (edge/transport) (G26–G32)

*(part of the [roadmap index](README.md))*

Ratings key: see [README.md § How to read the ratings](README.md#how-to-read-the-ratings).

**G26/G27/G29/G30 (+ the G48 rule in [scoring-fusion.md](scoring-fusion.md)
that depends on them) are a confirmed dead end as of 2026-07-21, not an open
infra backlog** — full finding in the G27 row below: Cloudflare's proxied mode
runs two fully independent connections at every layer (TCP, TLS, HTTP/2), so
no amount of origin-side engineering (custom nginx module, Go-terminated TLS
listener, raw packet capture) can expose the visitor's real network
characteristics — only Cloudflare's own edge-to-origin connection, identical
for every visitor. Closed; not a backlog item to revisit with more infra
investment.

| # | Capability they provide | Who has it | Sev · Effort · Status | What they do that we don't → recommended move |
|---|---|---|---|---|
| G26 | HTTP/2 frame fingerprint (Akamai-style) | BrowserScan.net, DataDome, bot.incolumitas | medium · high-infra · **Deferred (dead end, confirmed 2026-07-21)** | Fingerprint HTTP/2 setup: SETTINGS frame values, WINDOW_UPDATE, stream PRIORITY, pseudo-header ordering (Akamai hash format), folded by DataDome into JA4H. Distinguishes real browser h2 stacks from HTTP clients faking the UA. → **Same blocker as TLS (proxy terminates h2), and the same 2026-07-21 finding on G27 applies: a "JA4-capable edge" isn't a thing we can build ourselves here (see G27) — this closes with it, not a separate problem to solve.** |
| G27 | TLS ClientHello fingerprint (JA3/JA4) | bot.incolumitas, BrowserScan.net, DataDome, Pixelscan (pixelscan.dev) | medium · high-infra · Deferred (documented, concrete reason as of 2026-07-21) | Hash raw TLS ClientHello (cipher-suite list/order, extensions, curves, GREASE) into JA3/JA4, cross-check handshake matches browser UA claims (e.g. Chrome UA whose JA3 isn't Chrome's is a hard tell). BrowserScan surfaces JA3/JA3-hash/JA4 + cipher/extension detail; DataDome flags TLS-vs-UA class mismatch at edge before JS runs. → **2026-07-21: investigated building this ourselves (nginx `stream{}` + `ssl_preread` SNI routing to a Go-native `crypto/tls` listener for just this subdomain) and killed it before touching prod — verified the wrong blocker. Self-terminating TLS on our own nginx/Go doesn't work *even in principle* while Cloudflare proxy stays ON (required — it's what makes the `CF-Connecting-IP` trust model in [DEPLOYMENT.md](../../../../docs/DEPLOYMENT.md) §4 safe, and disabling it would also expose the shared origin IP for every other vhost behind the same nginx). Cloudflare terminates the visitor's TLS at its own edge and originates a *separate* TLS connection to origin — whatever ClientHello reaches nginx/Go there is **Cloudflare's own edge-to-origin handshake, identical for every visitor**, not the browser's. No amount of nginx/Go plumbing changes that; it would just measure "is this connection from Cloudflare," always yes. The only way to get the *real* visitor's JA3/JA4 while keeping proxy ON is Cloudflare's own edge-computed fingerprint, forwarded to origin as `cf-ja3-hash`/`cf-ja4` request headers — confirmed via [Cloudflare's own docs](https://developers.cloudflare.com/bots/additional-configurations/ja3-ja4-fingerprint/) that this is gated to **Enterprise customers who've purchased Bot Management**, not Free/Pro/Business. Not worth an enterprise-tier purchase for a personal tool. Reclassified alongside G26/G29/G30/G48: genuinely out of reach, not a backlog item to pick up later.** |
| G28 | DNS-leak and IPv6-leak tests | bot.incolumitas, BrowserScan.net, whoer.net, iphey.com | low · high-infra · **Not built** | Induce browser to resolve vendor-controlled hostnames, correlate resolver's IP/geo (+ any IPv6 path) against egress IP — DNS/IPv6 egressing outside proxy reveals real network. → **Skip. Requires controlled DNS infrastructure + unique per-session subdomains — heavy infra mostly serving anti-detect/VPN audience, not a bot self-test. Note as aware omission.** |
| G29 | HTTP header order / casing analysis | bot.incolumitas, deviceandbrowserinfo.com, DataDome | low · high-infra · **Deferred (dead end, confirmed 2026-07-21)** | Inspect order + casing of received HTTP headers (real browsers emit stable, characteristic order), flag HTTP-client-shaped ordering; DataDome folds header order into JA4H, detects browser-only headers being absent. → **Same root cause as G27, not just "nginx normalizes order" — even a Go-native listener for this subdomain would only see the header order Cloudflare's own edge chooses when re-issuing the request to origin, never whatever order the visitor's real browser sent. Closes alongside G27; not revisitable by adding infra.** |
| G30 | Passive TCP/IP SYN OS fingerprint (p0f/zardaxt) | bot.incolumitas, DataDome, whoer.net | low · high-infra · **Deferred (dead end, confirmed 2026-07-21)** | Passively infer real OS from SYN packet (TCP options, window size, IP fragmentation flag), cross-check vs claimed UA/OS. bot.incolumitas uses zardaxt.py; DataDome does Layer 3/4 fingerprinting at edge. → **Not just "no raw packet access" — even with raw packet capture at the origin, the SYN we'd see belongs to Cloudflare's own TCP stack opening a fresh connection to origin, not the visitor's OS/network stack. Structurally unfixable while Cloudflare proxy stays on. Closed.** |
| G31 | Proxy detection via latency triangulation | bot.incolumitas | low · high-infra · **Not built** | Compare browser-to-server RTT vs server-to-client-IP RTT; large asymmetry exposes proxy/VPN hop even when IP looks clean. → **Skip for now. Interesting but needs active server-initiated probing to client IP + careful timing; low incremental value over our existing IP2Proxy datacenter/VPN/Tor flags.** |
| G32 | Server-side open-port scan (22/3389) | bot.incolumitas, BrowserScan.net, whoer.net | low · high-infra · **Not built** | Scan connecting IP for open SSH (22) / RDP (3389) + other ports to reveal VPS/server/remote-desktop hosts betraying a non-consumer, automation-oriented environment. bot.incolumitas also treats a reachable CDP remote-debug port as a (weak) automation vector. → **Skip. Active outbound port-scanning from server is abuse-adjacent, slow, frequently blocked, off-brand for a stateless self-test page. Not worth building.** |
