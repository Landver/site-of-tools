# Roadmap — network-layer fingerprinting (edge/transport) (G26–G32)

*(part of the [roadmap index](README.md))*

Ratings key: see [README.md § How to read the ratings](README.md#how-to-read-the-ratings).
All blocked or deferred by same topology fact: nginx/Cloudflare terminate TLS
and normalize/downgrade HTTP in front of Go, so `crypto/tls` never sees raw
ClientHello and Echo never sees raw header order.

| # | Capability they provide | Who has it | Sev · Effort · Status | What they do that we don't → recommended move |
|---|---|---|---|---|
| G26 | HTTP/2 frame fingerprint (Akamai-style) | BrowserScan.net, DataDome, bot.incolumitas | medium · high-infra · Deferred (documented) | Fingerprint HTTP/2 setup: SETTINGS frame values, WINDOW_UPDATE, stream PRIORITY, pseudo-header ordering (Akamai hash format), folded by DataDome into JA4H. Distinguishes real browser h2 stacks from HTTP clients faking the UA. → **Documented as deferred. Same blocker as TLS (proxy terminates h2). If JA4-capable edge added for TLS, capture h2 fingerprint in same pass. Otherwise leave deferred; genuinely out of reach behind nginx.** |
| G27 | TLS ClientHello fingerprint (JA3/JA4) | bot.incolumitas, BrowserScan.net, DataDome, Pixelscan (pixelscan.dev) | medium · high-infra · Deferred (documented) | Hash raw TLS ClientHello (cipher-suite list/order, extensions, curves, GREASE) into JA3/JA4, cross-check handshake matches browser UA claims (e.g. Chrome UA whose JA3 isn't Chrome's is a hard tell). BrowserScan surfaces JA3/JA3-hash/JA4 + cipher/extension detail; DataDome flags TLS-vs-UA class mismatch at edge before JS runs. → **Our design docs already acknowledge this blocked because nginx/Cloudflare terminate TLS and crypto/tls hides ClientHello. Realistic path: run small TLS-passthrough listener (or use CF/nginx JA4 header exposure like ssl_preread / a uTLS-style sidecar) to capture JA4 and add one cross-check rule (JA4-implied browser vs claimed UA). High value as an unforgeable signal, but keep it explicitly optional so tool still runs behind a terminating proxy.** |
| G28 | DNS-leak and IPv6-leak tests | bot.incolumitas, BrowserScan.net, whoer.net, iphey.com | low · high-infra · **Not built** | Induce browser to resolve vendor-controlled hostnames, correlate resolver's IP/geo (+ any IPv6 path) against egress IP — DNS/IPv6 egressing outside proxy reveals real network. → **Skip. Requires controlled DNS infrastructure + unique per-session subdomains — heavy infra mostly serving anti-detect/VPN audience, not a bot self-test. Note as aware omission.** |
| G29 | HTTP header order / casing analysis | bot.incolumitas, deviceandbrowserinfo.com, DataDome | low · high-infra · Deferred (documented) | Inspect order + casing of received HTTP headers (real browsers emit stable, characteristic order), flag HTTP-client-shaped ordering; DataDome folds header order into JA4H, detects browser-only headers being absent. → **Deferred because nginx normalizes header order before Go sees it. If ever fronted by a Go-native TLS listener, Echo could read raw order. Low priority; stays blind as long as nginx is in front.** |
| G30 | Passive TCP/IP SYN OS fingerprint (p0f/zardaxt) | bot.incolumitas, DataDome, whoer.net | low · high-infra · Deferred (documented) | Passively infer real OS from SYN packet (TCP options, window size, IP fragmentation flag), cross-check vs claimed UA/OS. bot.incolumitas uses zardaxt.py; DataDome does Layer 3/4 fingerprinting at edge. → **Deferred, correctly so — requires raw packet access below the proxy/load balancer, which our container/edge topology doesn't grant. Keep on acknowledged-gap list; not worth the infra for a self-test tool.** |
| G31 | Proxy detection via latency triangulation | bot.incolumitas | low · high-infra · **Not built** | Compare browser-to-server RTT vs server-to-client-IP RTT; large asymmetry exposes proxy/VPN hop even when IP looks clean. → **Skip for now. Interesting but needs active server-initiated probing to client IP + careful timing; low incremental value over our existing IP2Proxy datacenter/VPN/Tor flags.** |
| G32 | Server-side open-port scan (22/3389) | bot.incolumitas, BrowserScan.net, whoer.net | low · high-infra · **Not built** | Scan connecting IP for open SSH (22) / RDP (3389) + other ports to reveal VPS/server/remote-desktop hosts betraying a non-consumer, automation-oriented environment. bot.incolumitas also treats a reachable CDP remote-debug port as a (weak) automation vector. → **Skip. Active outbound port-scanning from server is abuse-adjacent, slow, frequently blocked, off-brand for a stateless self-test page. Not worth building.** |
