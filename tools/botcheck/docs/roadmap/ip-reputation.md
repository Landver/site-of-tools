# Roadmap — IP reputation depth, crowd-blending & rarity (G36–G44)

*(part of the [roadmap index](README.md))*

Ratings key: see [README.md § How to read the ratings](README.md#how-to-read-the-ratings).

| # | Capability they provide | Who has it | Sev · Effort · Status | What they do that we don't → recommended move |
|---|---|---|---|---|
| G36 | Good-bot allowlist + AI-agent/LLM-crawler classification | DataDome, Fingerprint.com, BrowserScan.net | medium · low · **Shipped** | Distinguish verified good bots (Googlebot/Bingbot etc.) from malicious automation, and (Fingerprint/DataDome) classify known AI-company user agents to separate benign AI assistants/crawlers from bad bots. → **Shipped ([`goodbots.go`](../../goodbots.go)): a curated crawler + AI-agent allowlist, a `good-bot` verdict, and ASN-**number** corroboration (the crawler's own AS, distinct from any cloud the operator resells under the same name). The downgrade is granted ONLY for single-tenant operator ASNs (Apple/Yandex/Baidu/Naver/Seznam/Anthropic/Meta/ByteDance) an outsider can't originate from — multi-tenant clouds (Googlebot on AS15169, GPTBot on Azure) are recognised-but-unverified and still penalised, so no UA-spoof escapes. Full coverage of the cloud operators needs a published-IP-range check (follow-up).** |
| G37 | IP blacklist / DNSBL / abuser-score reputation | bot.incolumitas, BrowserScan.net, Pixelscan, whoer.net | medium · ml-or-db · **Not built** | Look up the egress IP against blacklists/DNSBLs and return an abuser_score / blacklist flag beyond mere datacenter/VPN/Tor classification. → **Our IP2Proxy PX12 gives datacenter/VPN/Tor/proxy but no reputation/abuser score. Adding a bundled DNSBL/reputation dataset (or an offline blocklist BIN) would strengthen the network tier without breaking statelessness. Medium value; check whether an IP2Location/IP2Proxy tier or a static blocklist can be bind-mounted like the existing BINs.** |
| G38 | Surface ASN/ISP and name the specific VPN/hosting provider _(we do a narrower version)_ | bot.incolumitas, Fingerprint.com, Pixelscan, whoer.net | low · low · **Shipped** | Name the ASN/ISP/company and the specific VPN service (bot.incolumitas identified NordVPN behind DataCamp/CDN77) rather than only a boolean datacenter/VPN flag. → **IP2Proxy/IP2Location records typically carry ISP/ASN/provider fields we may already have in the BINs but don't surface. Add these to the 'your request' card for transparency — cheap and improves the report without new data sources.** → **Shipped 2026-07-18 (the shared surface):** `platform.ConnInfo` gained additive `ASN`/`ASName`/`ProxyKind`/`ProxyProvider` fields + a `WithNetwork` enricher, and the conn partial renders guarded "ASN: AS12345 (Example ISP)" rows. → **Wired 2026-07-18 (wave 2):** botcheck's `check` handler (`POST /check`) now feeds the card from its iptools lookup result (`.WithNetwork(...)`), so the ASN/proxy rows render on the live page after a check runs. → **Wired 2026-07-18 (follow-up):** iptools' own handler enriches the card the same way on self-lookups (a `?ip=` lookup of someone else's IP never touches it), via the `Result.ConnNetwork()` mapping both tools now share. |
| G39 | Cross-customer / collective threat intelligence | DataDome | low · ml-or-db · **Not built** | Score an IP/fingerprint seen attacking one protected site across the entire customer network (network effect), plus a maintained known-bot signature repository grown by genetic algorithms. → **Not applicable — we're a single self-test page with no protected-site network. Note as an enterprise-only capability we intentionally don't pursue.** |
| G40 | Crowd-blending / fingerprint rarity / uniqueness entropy | CreepJS, Fingerprint.com, iphey.com, AmIUnique.org, EFF Cover Your Tracks, deviceandbrowserinfo.com, Pixelscan | low · ml-or-db · Deferred (documented) | Score a fingerprint against a visitor population: rarity/'one in X', Shannon-entropy bits per attribute, crowd-blending score with letter grades, or outlier detection against a real-people fingerprint DB — a rare/impossible fingerprint reads as fake. → **Deferred while botcheck stays stateless in practice. Requires a population corpus + storage; **MongoDB is now available** for the storage half, so accumulating a corpus and adding a minimal rarity table is the first crowd feature worth prototyping — but it's not a self-test priority.** |
| G41 | Fingerprint-reuse detection across requests | bot.incolumitas | low · ml-or-db · **Shipped** | Flag identical canvas/WebGL fingerprints repeated across many requests to unmask scraping-farm infrastructure (caught ScrapingBee returning a constant fingerprint). → **Shipped 2026-07-18 (wave 2):** the Mongo fingerprint corpus ([`corpus.go`](../../corpus.go), `botcheck_fingerprints` collection, 30-day TTL) records each POSTed fingerprint's stable-field hash plus the presenting IP; the `fingerprint_reuse` consistency rule (−25) fires at ≥5 distinct IPs in the window — the scraping-farm catch — and is suppressed for verified crawler fleets (G36), which legitimately share one fingerprint. |
| G42 | Fuzzy / locality-sensitive fingerprint hash + surfaced FP ID _(we do the exact ID)_ | CreepJS, incolumitas, Fingerprint.com | low · ml-or-db · **Shipped (exact ID)** | Compute both an exact fingerprint ID and a separate fuzzy/LSH hash so near-identical fingerprints cluster even when one attribute changes; surface the ID to the user. → **Shipped 2026-07-18 (wave 2, the exact-ID half):** `Signals.FingerprintHash()` — sha256 over the canonical stable client fields (UA, languages, platform, cores, memory, screen, timezone, WebGL, productSub, engine, font count) — is the exact fingerprint ID backing the corpus + the `fingerprint_reuse` rule (G41). The fuzzy/LSH near-duplicate half stays deferred: it only pays off alongside rarity scoring (G40). |
| G43 | Request velocity per device / IP over time windows | Fingerprint.com, DataDome | low · ml-or-db · **Not built** | Count distinct IPs / linked IDs per device (and requests per IP) over rolling windows to flag bursts and linkage. → **Needs cross-request state — bends the stateless rule; sits below the domain service, backed by MongoDB (already wired into botcheck for the G41/G42 fingerprint corpus, but not yet for a per-device/IP velocity counter) — see [internal-backlog.md](internal-backlog.md), Layer 2.** |
| G44 | Residential-proxy detection (distinct from datacenter/VPN) | Fingerprint.com, DataDome, Pixelscan | low · ml-or-db · **Shipped** | Detect residential proxies (graded confidence), the hard case aimed at agentic/AI fraud, separate from datacenter/VPN classification. → **PX12 may already tag some residential proxies; verify which proxy types the bundled BIN classifies and surface them. True residential-proxy detection at competitor quality needs a specialized feed we won't maintain — accept partial coverage.** → **Verified + shipped 2026-07-18 (the shared surface):** the bundled PX12 exposes exactly ten proxy-type codes (VPN, TOR, PUB, WEB, DCH, SES, RES, CPN, EPN, AIC — RES is the residential tag), and `platform.ProxyKindLabel` maps them to readable names the conn partial shows alongside the PX12 provider field ("Proxy: VPN — NordVPN" style). → **Wired 2026-07-18 with G38 (wave 2):** botcheck's `check` handler enriches the card from its iptools lookup result, so the mapped proxy kind + provider render on the live page after a check; iptools' own handler does the same on self-lookups (follow-up, same day) via the shared `Result.ConnNetwork()` mapping. |
