| G36 | Good-bot allowlist + AI-agent/LLM-crawler classification | DataDome, Fingerprint.com, BrowserScan.net | medium · low · **Shipped** | Distinguish verified good bots (Googlebot/Bingbot etc.) from malicious automation; Fingerprint/DataDome also classify known AI-company user agents to separate benign AI assistants/crawlers from bad bots. → **Shipped** ([`goodbots.go`](../../goodbots.go)): curated crawler + AI-agent allowlist, a `good-bot` verdict, ASN-**number** corroboration (crawler's own AS, distinct from any cloud operator resells under same name). Downgrade granted ONLY for single-tenant operator ASNs (Apple/Yandex/Baidu/Naver/Seznam/Anthropic/Meta/ByteDance) an outsider can't originate from — multi-tenant clouds (Googlebot on AS15169, GPTBot on Azure) recognised-but-unverified, still penalised, so no UA-spoof escapes. This is its own mechanism, not a `scoring.go` rule, so it has no `checks/` file of its own; it widened `bot_user_agent` (see [checks/bot_user_agent.md](../testing/checks/bot_user_agent.md)) and suppresses `fingerprint_reuse` for verified fleets (see [checks/fingerprint_reuse.md](../testing/checks/fingerprint_reuse.md)). | G37 | IP blacklist / DNSBL / abuser-score reputation | bot.incolumitas, BrowserScan.net, Pixelscan, whoer.net | medium · ml-or-db · **Not built** | Look up egress IP against blacklists/DNSBLs, return abuser_score / blacklist flag beyond mere datacenter/VPN/Tor classification. → **Our IP2Proxy PX12 gives datacenter/VPN/Tor/proxy but no reputation/abuser score. | G38 | Surface ASN/ISP and name the specific VPN/hosting provider _(we do a narrower version)_ | bot.incolumitas, Fingerprint.com, Pixelscan, whoer.net | low · low · **Shipped** | Name ASN/ISP/company + specific VPN service (bot.incolumitas identified NordVPN behind DataCamp/CDN77) rather than only a boolean datacenter/VPN flag. → **IP2Proxy/IP2Location records typically carry ISP/ASN/provider fields we may already have in the BINs but don't surface. | G39 | Cross-customer / collective threat intelligence | DataDome | low · ml-or-db · **Not built** | Score IP/fingerprint seen attacking one protected site across entire customer network (network effect), plus a maintained known-bot signature repository grown by genetic algorithms. → **Not applicable — we're a single self-test page w/ no protected-site network. Note as enterprise-only capability we intentionally don't pursue.** |
| G40 | Crowd-blending / fingerprint rarity / uniqueness entropy | CreepJS, Fingerprint.com, iphey.com, AmIUnique.org, EFF Cover Your Tracks, deviceandbrowserinfo.com, Pixelscan | low · ml-or-db · Deferred (documented) | Score fingerprint against a visitor population: rarity/'one in X', Shannon-entropy bits per attribute, crowd-blending score w/ letter grades, or outlier detection against a real-people fingerprint DB — rare/impossible fingerprint reads as fake. → **Deferred while botcheck stays stateless in practice. Requires population corpus + storage; **MongoDB now available** for storage half, so accumulating a corpus + adding a minimal rarity table is first crowd feature worth prototyping — but not a self-test priority.** |
| G41 | Fingerprint-reuse detection across requests | bot.incolumitas | low · ml-or-db · **Shipped** | Flag identical canvas/WebGL fingerprints repeated across many requests to unmask scraping-farm infrastructure (caught ScrapingBee returning a constant fingerprint). → **Shipped** — full implementation: [checks/fingerprint_reuse.md](../testing/checks/fingerprint_reuse.md), storage detail: [`../storage.md`](../storage.md). |
| G42 | Fuzzy / locality-sensitive fingerprint hash + surfaced FP ID _(we do the exact ID)_ | CreepJS, incolumitas, Fingerprint.com | low · ml-or-db · **Shipped (exact ID)** | Compute both an exact fingerprint ID and a separate fuzzy/LSH hash so near-identical fingerprints cluster even when one attribute changes; surface ID to the user. → **Shipped (exact-ID half)** — same implementation as G41: [checks/fingerprint_reuse.md](../testing/checks/fingerprint_reuse.md). Fuzzy/LSH near-duplicate half stays deferred: only pays off alongside rarity scoring (G40). |
| G43 | Request velocity per device / IP over time windows | Fingerprint.com, DataDome | low · ml-or-db · **Not built** | Count distinct IPs / linked IDs per device (and requests per IP) over rolling windows to flag bursts + linkage. → **Needs cross-request state — bends the stateless rule; sits below the domain service, backed by MongoDB (already wired into botcheck for the G41/G42 fingerprint corpus, but not yet for a per-device/IP velocity counter) — see [internal-backlog.md](internal-backlog.md), Layer 2.** |
