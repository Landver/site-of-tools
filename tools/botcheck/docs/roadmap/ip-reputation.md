| G37 | IP blacklist / DNSBL / abuser-score reputation | bot.incolumitas, BrowserScan.net, Pixelscan, whoer.net | medium · low · **Shipped (ipsum, 2026-07-21)** | Look up egress IP against blacklists/DNSBLs, return abuser_score / blacklist flag beyond mere datacenter/VPN/Tor classification. → **Shipped** as the `ip_blocklisted` rule (consistency/network, weight 25) reading a new **shared** Mongo collection `ip_blocklist` — see [checks/ip_blocklisted.md](../testing/checks/ip_blocklisted.md) and [storage.md](../storage.md). NB: implemented as a **Mongo-backed shared corpus, not the bind-mounted flat file** the research first proposed — the collection is a common blocklist any service/script/workflow can write into (fields: ip, source, reason, count, meta, created_at, updated_at; unique `(ip, source)`; 60-day TTL on `updated_at`), so ipsum is just its first writer and botcheck its first reader. Second layer, not yet added: Spamhaus DROP/EDROP (free-for-commercial-use hijacked-netblock list, attribution-only like IP2Location LITE) — the tool owner emailed Spamhaus about the narrow ToU §3.1 IP-licence gap; add once confirmed. CINS Army list held as a later nice-to-have (best-in-class freshness, a genuinely different scanner/RDP/C2 signal) pending a written-permission email to its maintainer, since it rests on an informal use-it-freely statement rather than any actual licence. **Ruled out:** FireHOL/blocklist-ipsets (repo itself carries no licence, and silently bundles AbuseIPDB's no-resale-restricted files), Team Cymru Bogons (wrong signal — allocation status, not abuse), Tor Project's list + X4BNet's VPN/proxy/Tor lists (redundant with PX12), AbuseIPDB (explicitly bans commercial use + resale on the free tier, and is API-only, not a static file, anyway), GreyNoise Community API (API-only with a EULA that plausibly bars building a public tool on the free tier), MaxMind GeoIP2 Anonymous IP DB (not free, and MaxMind's own docs explicitly list "public visitors... geolocate IP addresses" as a prohibited use case — a direct match to what botcheck would be doing), abuse.ch feeds (the only IP-relevant feeds among Feodo Tracker/SSLBL/URLhaus are stale or have been quietly deprecated since 2025). blocklist.de sits on the bubble — good complementary signal, well maintained, but carries an active "written consent required" reproduction clause the verification pass couldn't rule out covering the IP lists; worth a parallel permission email but ranked behind CINS Army.** |

| G40 | Crowd-blending / fingerprint rarity / uniqueness entropy | CreepJS, Fingerprint.com, iphey.com, AmIUnique.org, EFF Cover Your Tracks, deviceandbrowserinfo.com, Pixelscan | low · ml-or-db · Deferred (documented, with a concrete reason as of 2026-07-21) | Score fingerprint against a visitor population: rarity/'one in X', Shannon-entropy bits per attribute, crowd-blending score w/ letter grades, or outlier detection against a real-people fingerprint DB — rare/impossible fingerprint reads as fake. → **Deferred as a *scoring* signal — and 2026-07-21 gave the deferral a concrete reason rather than "not a priority": rarity does **not** discriminate bots at a self-test tool's scale. Almost every visitor is new, so their exact fingerprint is *maximally rare* (seen once) — which describes a first-time human and a bespoke automation client identically. Rarity only separates the two against a large population where per-attribute frequencies are known ("your GPU is 1-in-10 000"), and computing those requires storing per-attribute fingerprint detail per visitor (not the current one-way full-fingerprint hash) — a privacy cost not worth paying for a signal that doesn't discriminate. So the corpus's discriminating signals were built instead — `fingerprint_reuse` (spatial, G41/G42) and now `ip_fingerprint_churn` (temporal, G43) — while rarity/entropy stays deferred as *scoring*. A rarity/'one in X' *informational* readout (à la AmIUnique/EFF, explicitly not a verdict) remains a possible future transparency feature, gated on the same per-attribute storage decision. See G58 in [reporting-ux.md](reporting-ux.md).** |

| G43 | Request velocity per device / IP over time windows | Fingerprint.com, DataDome | low · ml-or-db · **Shipped (churn variant)** | Count distinct IPs / linked IDs per device (and requests per IP) over rolling windows to flag bursts + linkage. → **Shipped 2026-07-21 as `ip_fingerprint_churn`** — the temporal inverse of `fingerprint_reuse`, on the same `botcheck_fingerprints` corpus: `Corpus.DistinctHashesByIP(ip, 10-min window)` counts how many **distinct fingerprints** one egress IP cycled through, firing at ≥8.
